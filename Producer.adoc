== Назначение

Сервис заказов (Dispute Service) публикует событие `DecisionMakingEvent` в Kafka после принятия решения Арбитром.
Потребителями события являются:

. Сервис сделок (измение статуса сделки)
. Сервис работы с оплатами (выполнение возврата ранее заблокированных средств покупателю)
. Сервис уведомлений (отправка уведомления покупателю и продавцу о решении)
. Сервис аудита (сохранение события в логе)

== Топик

|===
| *Свойство*             | *Значение*
| Имя топика             | `DecisionMakingByAdmin`
| Количество партиций    | 4
| Ключ партиционирования | `orderId` (строка)
| Срок хранения          | 7 дней
| Фактор репликации      | 1
|===

== Схема события (JSON)

Событие содержит полную информацию о принятом решении Арбитром по спору определенной сделки.

[source,json]
----
{
  "eventId": "UUID",
  "dealId": "12345",
  "disputeId": "6789",
  "statusDecision": "RESOLVED_BUYER_WINS",
  "trackNumber": "202604-2048T12",
  "totalAmount": 12500.00,
  "currency": "RUB"
}
----

== Конфигурация продюсера

- **bootstrap.servers**: кластер Kafka (например, `kafka-cluster:9092`)
- **acks**: `all` — подтверждение от всех реплик
- **retries**: `5` — количество повторных попыток при временной ошибке
- **enable.idempotence**: `true` — гарантия уникальности сообщений в пределах продюсера

== Логика работы продюсера

1. Сервис споров формирует событие `DecisionMakingEvent`, заполняя все поля.
2. Событие отправляется в Kafka с ключом `disputeId` (для сохранения порядка сообщений по каждому спору).
3. Отправка выполняется асинхронно, не блокируя основной поток обработки споров.
4. Продюсер ожидает подтверждения от брокера (`acks=all`). Если подтверждение не получено, выполняется повторная попытка (до 5 раз) с экспоненциальной задержкой.
5. При успешной отправке фиксируется метаинформация (партиция, смещение) и событие считается опубликованным.
6. При неудаче после всех попыток:
- Событие сохраняется в таблицу `failed_events` в базе данных.
- В лог записывается ошибка с указанием `eventId`, `dealId` и `disputeId`.
- Для мониторинга увеличивается счётчик ошибок.

== Обработка ошибок и восстановление

- **Автоматические повторы**: Продюсер сам выполняет повторные попытки в случае временных сетевых сбоев.
- **Постоянное хранение неудачных событий**: Отдельный шедулер каждые 5 минут сканирует таблицу `failed_events` и повторяет отправку.
- При успешной повторной отправке запись удаляется из таблицы.
- После трёх неудачных повторных попыток запись помечается статусом `MANUAL_REQUIRED` и отправляется оповещение администратору.
- **Требования к идемпотентности**: Поскольку продюсер использует идемпотентность (`enable.idempotence=true`), повторные отправки одного и того же события (например, после сбоя) не приведут к дублированию в партициях.

== Мониторинг

. **Метрики**:
.. `dispute_events_producer_success_count` — количество успешно отправленных событий
.. `dispute_events_producer_failure_count` — количество неудачных отправок
.. `dispute_events_producer_retry_count` — количество повторных попыток
. **Оповещения**:
.. При превышении доли ошибок >1% за 5 минут
.. При появлении записи в `failed_events` со статусом `MANUAL_REQUIRED`
